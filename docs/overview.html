AMA-MIDI

A real-time collaborative MIDI sequencer for Amanotes — a shared visual workspace where composers, game developers, QA, and product owners can place, review, and iterate on music note sequences together, without corrupting shared data.

React 18 + Vite NestJS PostgreSQL 15 Redis 7 Socket.io Turborepo monorepo Anthropic Claude API
Rule 1
Visual Clarity
10,000 notes must be scannable and interactive — never frozen or laggy
Rule 2
Data Integrity
No two notes can occupy the same (track, time) position — enforced at DB level, not app level
Rule 3
Real-time Sync
Changes from any collaborator propagate to all open tabs within milliseconds

⚙️Functional Requirements

🔐
Authentication
FR-01 to FR-04
  • Login via Google OAuth (SSO) — no password management
  • On first login, create user record with role=COMPOSER
  • JWT returned — all API calls require valid JWT
  • Roles enforced: ADMIN, COMPOSER, VIEWER
  • JWT refresh on WebSocket reconnect (httpOnly refresh token)
🎵
Song Management
FR-05 to FR-09
  • Create song with name
  • List all songs with creator, note count, track activity
  • Open song into editor
  • Rename song (creator or ADMIN)
  • Delete song (ADMIN only, soft delete)
  • Create/list/restore named version snapshots
📍
Note Management
FR-10 to FR-18
  • Create note: track(1–8), time(0–300s, 0.1s snap), title, color
  • Duplicate position → 409 POSITION_TAKEN (never 500)
  • Update: title, description, color only (position immutable)
  • Delete: soft delete via deletedAt timestamp
  • Time rounded to 1 decimal before insert
  • Every mutation emits domain event (ledger + WS)
  • Fetch by time range: ?timeFrom=X&timeTo=Y
⚡
Real-time Collaboration
FR-19 to FR-24
  • WebSocket room per song (join/leave)
  • All note mutations broadcast to room
  • Presence: user joined/left events
  • Redis Pub/Sub bridges multiple API instances
  • Frontend updates TanStack Query cache on WS events (no re-fetch)
  • Reconnect with refreshed JWT on token expiry
📜
Change History + Undo
FR-25 to FR-30
  • Every note mutation writes NoteEvent (append-only)
  • Events have: before_state + after_state JSONB
  • History panel: paginated, newest-first, human language
  • Undo: compensating NOTE_DELETED event
  • Undo broadcasts to all collaborators via WS
  • Edge case: note already deleted by collaborator → graceful message, no crash
🤖
AI Note Suggester
FR-31 to FR-35
  • Available only to COMPOSER role
  • Enabled after 5+ notes placed
  • Sends last 10 notes to Claude API
  • Returns 4 suggestions: track, time, color
  • Suggestions render as ghost notes (20% opacity, dashed, pulse)
  • Accept → normal POST flow (conflict handling applies)
  • Dismiss → removes ghost locally

🎯Use Cases

UC-01 Composer places a note in Fast Mode Actor: Composer
1Composer clicks a position on the piano roll grid
2System calculates track (from X position) and time (from Y position, snapped to 0.1s)
3Ghost note appears at that position immediately (optimistic UI)
4POST /songs/:id/notes fires in background
5a ✓Success: ghost replaced with server note; NoteEvent written; all tabs receive note-created via WebSocket
5b ✗409 POSITION_TAKEN: ghost disappears, toast "This position was just taken — try a nearby spot"
UC-02 Two composers write to the same position simultaneously Actor: Composer A + Composer B
1Both A and B click the same (track, time) position at the same moment
2Both see ghost notes on their screen immediately
3Both POSTs hit the API simultaneously
4PostgreSQL unique constraint: one INSERT wins, one gets P2002 violation
5aWinner (201): note saved, NoteEvent written, broadcast to all tabs
5bLoser (409): ghost disappears, toast shown. DB has exactly 1 note.
UC-03 Composer undoes last note Actor: Composer
1Composer clicks Undo in History Panel
2POST /songs/:id/events/undo fires
3Backend finds last NOTE_CREATED event by current user
4aNote exists: soft-delete it, write NOTE_DELETED compensating event, broadcast note-deleted via WS
4bNote already deleted by collaborator: write NOTE_DELETED with alreadyDeleted:true, return friendly message
4cNo notes to undo: return NOTHING_TO_UNDO, show toast "Nothing to undo"
UC-04 Composer requests AI note suggestions Actor: Composer (≥5 notes placed)
1Composer clicks "Suggest" button (enabled after 5+ notes)
2POST /songs/:id/suggest-notes — backend fetches last 10 notes, calls Claude API
3Claude returns 4 suggestions [{track, time, color}], validated server-side
4Ghost notes appear on grid (20% opacity, dashed border, pulse animation)
5aAccept: POST /notes → normal create flow, becomes real note, broadcasts to all tabs
5bDismiss: ghost removed from local state only
UC-05 QA engineer validates a song Actor: QA Engineer
1QA switches to QA View mode via toolbar chip
2Frontend calls GET /songs/:id/validation
3Backend ValidationEngine runs 4 rules: boundary, density, gap, empty-track
4Issues returned as [{ruleId, severity, noteId, time, track, message}]
5Affected notes get orange/yellow ring highlights on grid
6Validation panel always open on right, "J" key jumps between issues

JWT Expiry Edge Case

When the JWT expires during an active session, Socket.io reconnect will fail with "jwt expired". The useSocket hook catches this, calls POST /auth/refresh (httpOnly refresh token cookie), gets a new access token, and retries the connection automatically. If the refresh token is also expired → redirect to login with "Your session expired" toast.

🚨Architecture Invariants

These rules must never be violated. Breaking any one creates a class of bugs that cannot be fixed without a rewrite.

Invariant 1 — Decoupling
NoteService has ZERO imports from LedgerModule or RealtimeModule
Uses EventEmitter2 only. Side-effects are handled by @OnEvent listeners. Adding new consumers (validation, analytics) never touches NoteService.
Invariant 2 — CQRS Split
NoteService has ZERO read methods — all reads go through NoteQueryService
Writes care about integrity; reads care about performance. Separate classes allow independent optimization and future read replicas.
Invariant 3 — DB-Level Integrity
Duplicate prevention is enforced at DB layer ONLY — no app-level pre-check as sole guard
Application-level "check then insert" has a race condition under concurrent writes. The PostgreSQL unique index is atomic. P2002 → 409.
Invariant 4 — Event-Driven Mutations
Every note mutation must emit a domain event — ledger and WebSocket are @OnEvent listeners
Ledger writes and WS broadcasts are side-effects of note mutations, not direct calls. Guarantees consistent event log.
Invariant 5 — Soft Deletes Only
Notes use deletedAt TIMESTAMPTZ — never DELETE FROM notes
The NoteEvent ledger references note IDs. Hard deleting notes would break foreign keys and orphan event history.
Invariant 6 — Pure Engine Layer
All coordinate math lives in features/editor/engine/ — pure TypeScript, zero React imports
If xToTrack(), yToTime(), getVisibleTimeRange() live in React components, they're untestable and get copy-pasted across 6+ places. One source of truth.
Invariant 7 — WebSocket is Notification-Only
All writes go through REST API → DB → EventEmitter → WebSocket. Never write directly to DB from WebSocket handlers.
WebSocket handlers are fire-and-forget. DB writes require error handling, validation, and transaction guarantees that REST controllers provide.
